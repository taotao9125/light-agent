# 5.6 - 5.12 学习与实现目标

## 总目标

这一周的目标是把 MCP、workflow、agent 从概念落到可运行代码：

```text
MCP client
  -> MCP server
    -> tools
      -> workflow runtime
        -> state store
        -> effects
        -> external events
```

重点不是一次性做完整框架，而是每天完成一个能跑、能解释、能继续扩展的小闭环。

## Day 1：看懂 workflow 核心执行模型，并自己写出最小 runtime

### 执行任务

- 重新阅读 `server/demos/min-workflow-yield.js`，只关注主流程，不追求一次看懂所有细节。
- 自己从 0 写一个更小的 workflow demo。
- 最小支持 `call` 和 `wait` 两类 effect。
- 使用本地 JSON 文件保存 workflow state。
- 实现以下生命周期：

```text
reset
run 第一次
approve
run 第二次
```

### 验收标准

- 能通过命令完成完整流程：

```bash
node demos/your-workflow.js reset
node demos/your-workflow.js
node demos/your-workflow.js approve
node demos/your-workflow.js
```

- 第一次 `run` 能执行到等待点并退出。
- `approve` 能写入外部事件。
- 第二次 `run` 能从上次状态恢复并完成。
- 能解释清楚：

```text
state 存在哪里
currentStep 表示什么
为什么已经完成的 step 不会重复执行
wait 为什么能让流程暂停
approve 为什么能让流程恢复
```

## Day 2：实现最小 MCP server/client，让协议通信先跑起来

### 执行任务

- 不使用 MCP SDK，手写最小 MCP client/server。
- 先选择一种 transport，优先选择当前更容易理解的 HTTP。
- 实现最小 JSON-RPC 结构：

```text
jsonrpc
id
method
params
result
error
```

- server 最小支持：

```text
initialize
tools/list
tools/call
```

- 只实现一个 tool：

```text
add(a, b)
```

### 验收标准

- 启动 server 后，client 能完成完整调用。
- client 能打印：

```text
initialize result
tools/list result
tools/call result
```

- `tools/call` 调用 `add` 后能返回正确结果。
- 能解释清楚：

```text
MCP message 和普通 HTTP request 的区别
tools/list 的作用
tools/call 的作用
tool arguments 从哪里来
tool result 怎么返回
```

## Day 3：把 workflow runtime 接到 MCP tool 后面

### 执行任务

- 保留 Day 1 的 workflow runtime。
- 在 Day 2 的 MCP server 中新增 workflow 相关 tools。
- 最小 tools：

```text
workflow_reset
workflow_run
workflow_approve
workflow_state
```

- MCP client 通过 `tools/call` 操作 workflow。

### 验收标准

- client 能按顺序调用：

```text
workflow_reset
workflow_run
workflow_approve
workflow_run
workflow_state
```

- 第一次 `workflow_run` 返回 `waiting`。
- `workflow_approve` 后，第二次 `workflow_run` 返回 `completed`。
- `workflow_state` 能看到完整状态。
- 能解释清楚：

```text
MCP server 只是入口
workflow runtime 负责状态和执行
tool handler 如何调用 workflow runtime
```

## Day 4：实现一个最小 agent loop，不接真实 LLM，先跑通流程控制

### 执行任务

- 写一个最小 `agent.js` 或 `agent.ts`。
- 暂时不接真实 LLM。
- agent 根据 workflow state 做固定规则判断。
- 最小状态判断：

```text
not_started -> reset/run
running -> run
waiting -> 提示需要 approval
completed -> 输出总结
failed -> 输出失败原因
```

### 验收标准

- agent 能启动 workflow。
- workflow 等待审批时，agent 能识别 `waiting` 状态。
- 外部执行 approve 后，agent 再次运行能恢复 workflow。
- workflow 完成后，agent 能输出最终状态。
- 能解释清楚：

```text
agent loop 是什么
agent 和 workflow runtime 的边界是什么
为什么 agent 不应该直接保存所有执行状态
```

## Day 5：接入真实 LLM 或模拟 LLM 决策，让它根据状态选择下一步

### 执行任务

- 把 Day 4 的固定规则决策抽成 `decideNextAction(state)`。
- 可以先用模拟 LLM 返回结构化 action。
- 如果条件允许，再接真实 LLM。
- LLM 或模拟决策只输出 action，不直接执行副作用。
- action 最小集合：

```text
start_workflow
resume_workflow
ask_user_approval
show_completed
show_failed
```

### 验收标准

- 输入不同 workflow state，能得到不同 action。
- action 由 agent 执行，而不是由 LLM 直接执行。
- 能解释清楚：

```text
LLM 负责决策
agent loop 负责调度
workflow runtime 负责可靠执行
tool 负责具体副作用
```

## Day 6：补上失败、重试、补偿这些真实执行场景

### 执行任务

- 给 workflow runtime 增加失败场景。
- 至少实现一个可重试步骤。
- 至少实现一个 fallback 或补偿逻辑。
- 模拟副作用已经发生后失败的场景。
- 记录失败原因到 workflow state。

### 验收标准

- 某个 step 第一次失败后可以 retry。
- retry 失败后可以进入 fallback 或 failed 状态。
- 创建 booking 后发生 fatal error 时，可以执行补偿逻辑。
- state 中能看到：

```text
failed step
retry count
last error
compensation result
final status
```

- 能解释清楚：

```text
为什么有副作用的 step 不能随便重复执行
retry 和 resume 的区别
compensation 解决什么问题
```
